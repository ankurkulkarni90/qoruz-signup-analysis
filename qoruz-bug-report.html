Google OAuth Signup Broken on Qoruz

Complete signup path failure with zero error feedback. OAuth flow completes on Google's side, but Qoruz's backend rejects the request silently. Every user who clicks "Sign up with Google" hits a dead end.

Section 01

Summary

Google OAuth signup on Qoruz is completely non-functional. The OAuth flow completes successfully on Google's side (consent granted, popup closes), but Qoruz's frontend sends the resulting JWT to /api/register without the required gReCaptchaToken field. The backend returns a 422 Unprocessable Entity, the frontend silently swallows the error, and the user sees no feedback whatsoever. The page simply stays on the signup form as if nothing happened.

This is not a client-side or browser-specific issue. It is a server-side validation misconfiguration combined with a frontend error-handling gap.

Impact

100% of users who attempt Google OAuth signup will fail silently. No error is shown, no lead data is captured, and no recovery path is offered.

Section 05

Root Cause Analysis

Primary Cause: Missing reCAPTCHA Token in Google OAuth Payload

The POST /api/register request sent by the frontend after Google OAuth completion contains the following payload:

Request Payload: OAuth Signup
{
  "oauthResponse": "<Google JWT>",
  "is_google_login": true,
  "utm_source": "SEO",
  "utm_medium": "organic search",
  "lead_source": "Inbound - SEO",
  "lead_page": "signup"
}

The backend returns:

Response: 422 Unprocessable Entity
{
  "ok": false,
  "error": "validation_failed",
  "validation_errors": {
    "gReCaptchaToken": [
      "The g re captcha token field is required."
    ]
  },
  "stuff": "The given data was invalid."
}

The backend requires gReCaptchaToken as a mandatory field on every /api/register call. The manual signup form includes a reCAPTCHA widget that generates and attaches this token. The Google OAuth signup path bypasses the reCAPTCHA widget entirely and never includes the token in its payload. The backend does not differentiate between OAuth and manual signups in its validation rules.

Secondary Cause: Silent Error Handling on Frontend

When the /api/register call returns a 422, the frontend does not surface any error to the user. No modal, no toast, no inline error message, no console-visible user-facing error. The user is left staring at an unchanged signup form with no indication that something went wrong. The only evidence is in the browser's Network tab (the failed register request) and Console (a POST 422 error line buried under hundreds of carousel warnings).

Contributing Issue: COOP Header Blocking postMessage

The browser console logs the following during the OAuth callback:

Cross-Origin-Opener-Policy policy would block the window.postMessage call.

This indicates that the Cross-Origin-Opener-Policy headers on app.qoruz.com are interfering with the communication channel between the Google OAuth popup and the parent page. While the OAuth token does eventually reach the parent page (evidenced by the /api/register call being made), this COOP misconfiguration could cause intermittent failures or race conditions in the popup-to-parent handoff.

Contributing Issue: Duplicate GSI Initialization

[GSI_LOGGER]: google.accounts.id.initialize() is called multiple times.

Google's Sign-In library is being initialized more than once, likely triggered by the testimonial carousel's continuous re-rendering (the carousel fires setState in an infinite loop due to a missing showThumbs={false} prop). This could cause unpredictable behavior in the OAuth callback registration.

Section 08

Additional Observations

Work Email Enforcement Gap

Even if the reCAPTCHA issue were fixed, Google OAuth signup using public email domains (Gmail, Yahoo, etc.) would hit a second wall. Qoruz requires a "work email" for registration. The manual form surfaces a clear modal for this restriction. The Google OAuth path has no equivalent handling. It would receive the Gmail address from the JWT, attempt to register, and presumably fail at the email domain validation stage (which currently never fires because the reCAPTCHA validation fails first).

Carousel Rendering Loop

The testimonial carousel on the signup page fires renderThumbs errors in an infinite loop. Each cycle triggers a setState call, causing continuous re-renders. The console accumulates hundreds of warnings within seconds. The fix: set showThumbs={false} on the Carousel component.

Mixed Content Warnings

The page loads an asset over HTTP from an HTTPS page (http://terareach.s3.ap-southeast-1.amazonaws.com/...). Chrome auto-upgrades this, but it indicates the S3 bucket references haven't been updated to use HTTPS URLs.

Section 09

Recommended Fixes

FIX 1

Resolve reCAPTCHA Validation for OAuth Signups

Option A (Backend, recommended): Make gReCaptchaToken conditionally required. When is_google_login is true and a valid Google JWT is present, skip the reCAPTCHA validation. The OAuth flow itself provides equivalent bot-protection assurance.

Option B (Frontend): Execute an invisible reCAPTCHA (grecaptcha.execute()) during the OAuth callback, before sending the /api/register request. This requires no backend changes but adds latency.

FIX 2

Surface Errors to the User

The frontend must handle non-2xx responses from /api/register and display meaningful feedback. At minimum, show a toast or modal with a generic "Something went wrong" message. Ideally, parse the error response and display actionable guidance.

FIX 3

Handle Work Email Enforcement for OAuth Signups

For Google OAuth signups where the returned email is a public domain, either prompt the user for a work email after OAuth completes (pre-filling their name from the JWT) or display the same "Work email required" modal shown on the manual form path.

FIX 4

Fix COOP Headers for OAuth Popup Communication

Review the Cross-Origin-Opener-Policy response headers. The current policy blocks window.postMessage calls from the Google OAuth popup. Adjust to same-origin-allow-popups to permit the OAuth callback communication.

FIX 5

Stop Duplicate GSI Initialization

Ensure google.accounts.id.initialize() is called exactly once, outside of any component render cycle.

FIX 6

Fix the Carousel Rendering Loop

Add showThumbs={false} to the react-responsive-carousel Carousel component on the signup page, or provide actual image elements for the thumbnail renderer.

FIX 7

Update S3 Asset URLs to HTTPS

Replace http://terareach.s3.ap-southeast-1.amazonaws.com/... references with https:// equivalents to eliminate mixed content warnings.
